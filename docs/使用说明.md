# 使用说明

## 1. 项目目标

这个项目把你现在在用的两段流程整理成了一个独立仓库：

1. 先把原始 GWAS summary statistics 标准化成统一格式。
2. 再把标准化后的 exposure 和 outcome 送入 `TwoSampleMR` 流程做单暴露单结局 MR。

## 2. 核心脚本

- [scripts/gwas_standardizer.py](/home/ding/MR_GWAS_Pipeline/scripts/gwas_standardizer.py)
  - 负责标准化、QC、canonical 位点 ID 生成
  - `--mr-role exp` 时会自动 clump
  - 交互模式下会进一步询问 clump 参数
- [scripts/mr_format_and_clump.R](/home/ding/MR_GWAS_Pipeline/scripts/mr_format_and_clump.R)
  - 负责 `format_data()` 和 exposure clump
- [scripts/MR_single_pipeline.R](/home/ding/MR_GWAS_Pipeline/scripts/MR_single_pipeline.R)
  - 负责 `harmonise_data()`、`MR-PRESSO`、`mr()`、`RAPS` 和出图

## 4. 推荐工作流

### 第一步：安装 Python 依赖并准备 R 包

```bash
cd /home/ding/MR_GWAS_Pipeline
bash bin/bootstrap.sh
```

R 包请根据 [config/r_packages.txt](/home/ding/MR_GWAS_Pipeline/config/r_packages.txt) 自行安装。

### 第二步：保存原始 GWAS 文件

建议把下载的原始 GWAS 文件放进：

```text
data/Org/
```

这样项目结构会更清楚，交互模式下输入路径也会优先指向这个目录。

### 第三步：生成 outcome 文件

把标准化后的 outcome 写到 `data/out/` 下，例如：

```bash
bash bin/standardize_gwas.sh \
  --non-interactive \
  --input /path/to/outcome.tsv.gz \
  --output data/out/HF.csv \
  --output-format mr \
  --mr-role out \
  ...
```

如果不提供 `--output`，脚本会把文件写到 `OUT_OUTPUT_DIR`。

### 第四步：生成 exposure 文件

把标准化后的 exposure 写到 `data/exp/` 下，例如：

```bash
bash bin/standardize_gwas.sh \
  --non-interactive \
  --input /path/to/exposure.tsv.gz \
  --output data/exp/EXPO.csv \
  --output-format mr \
  --mr-role exp \
  ...
```

如果不加 `--non-interactive`，脚本会交互式询问：

- 输出格式
- MR 角色
- phenotype
- 样本量
- clump 参数
- `plink` 路径
- clump 用的 `bfile` 前缀

如果加了 `--non-interactive`，这些值则默认从 [config/defaults.env](/home/ding/MR_GWAS_Pipeline/config/defaults.env) 或命令行参数读取。

### 第五步：运行 MR

```bash
bash bin/run_mr.sh EXPO HF
```

其中：

- `EXPO` 对应 `data/exp/EXPO.csv`
- `HF` 对应 `data/out/HF.csv`

## 5. 输出说明

### 标准化输出

MR-ready 输出里最关键的是三列：

- `SNP`
  - 实际存的是 `bim_id`
  - 用于 `clump_data()` 和后续 `harmonise_data()` 匹配
- `variant_id`
  - `CHR:POS:A1:A2`
  - `A1/A2` 按字典序排序
- `rsid`
  - 注释信息

标准化 TSV 还会额外保留：

- `ALLELE1` / `ALLELE2`
  - 排序后的两条等位基因
- `EFFECT_ALLELE` / `OTHER_ALLELE`
  - `beta` 实际对应的效应方向
- `EAF` / `MAF`
  - 如果输入只有 `MAF`，无 reference 模式下 `EAF` 可能为空

### MR 结果输出

每次分析会在 `results/<exposure>_<outcome>/` 下生成：

- `report.txt`
- `mr_results.txt`
- `iv_table.txt`
- `heterogeneity.txt`
- `pleiotropy.txt`
- `scatter.png`
- `funnel.png`
- `leaveoneout.png`

## 6. 已知事项

### 无 reference 模式的边界

- `Mode C (A1/A2)` 必须明确告诉脚本效应值对应 `A1` 还是 `A2`，`Unknown` 不再支持。
- 如果输入频率列是 `MAF` 而不是效应等位基因频率，脚本不会强行把它转成 `EAF`。
- 不再做 reference-based strand flip、回文位点消歧和方向修正。
- 因此更适合已经明确提供 `effect allele / other allele` 或已知 `A1/A2` 方向的数据。

### R 包管理

项目不再自动安装 R 包，只维护一个清单文件 [r_packages.txt](/home/ding/MR_GWAS_Pipeline/config/r_packages.txt)。

### 运行时间

- 标准化全量 GWAS 时，主要耗时来自文件读取和 `TwoSampleMR`/`MR-PRESSO`。
- MR 步骤里最慢的是 `MR-PRESSO`，IV 越多越慢。

### R 版本警告

当前机器默认 `Rscript` 是 `4.3.1`，而你的用户库目录是 `/home/ding/R/4.4.1_MR`。现在实际可运行，但可能会看到 package build version warning。这类 warning 不影响当前已验证过的流程运行。
